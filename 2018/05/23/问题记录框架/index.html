问题记录框架

By Michael

Published 2018-05-23

平时开发工作过程中，总会遇到一些值得我们反思和总结的问题。把这些经验教训以文档的形式记录下来，形成文档存档到知识库中，是一件非常有价值并值得去做的事情。大家都会做总结或记录，然而，如果没有统一的一种文档规范格式，大家的总结记录文档的内容组织格式可能多种多样。为了形成更统一、清晰的问题记录文档，并经过一段时间的实际使用，总结出以下的问题总结文档框架结构。

问题总结文档框架结构：

  • 问题名称
  • 问题现象
  • 影响范围
  • 原因分析
  • 解决方案
  • 实施策略

这是个完整的框架结构，并非每个问题都需要描述上述框架的各个部分，需根据具体问题而定。但是，以下基本的这些部分必须包含：

  • 问题名称
  • 问题现象
  • 原因分析
  • 解决方案

这四个部分是一个问题总结文档中必须包含的要素，不可或缺。

当积累的问题记录文档多了以后，如何按关键字快速、方便查找相关文档成为一个必须考虑的问题，为此，需有统一的文档命名规则，方便后续的查找，命名规则为：yyyyMMdd-问题名称.md，文件名中包含了问题记录的日期和问题关键字。

问题名称

给该问题取一个具有描述性的名称，简短精炼，包含问题核心关键字。

问题现象

描述问题发现的日期，该问题由谁那边发现和反馈的，并详细描述问题表现出来的现象。此处重点在于描述问题的现象，要如实而准确的记录，不要做假设性描述。

影响范围

评估该问题可能造成的影响范围，比如核实从哪个软件版本引入该问题，该问题的严重程度，涉及影响的终端设备批次和数量是多少，等等。

原因分析

详细描述问题的分析过程。通过收集相关数据、分析、验证，找出问题的真正原因，并以准确的语言条理清晰地描述出问题产生的原因。

解决方案

描述针对该问题的具体解决方案。需要给出该问题可能的一些解决方案，以及每个解决方案的优缺点。对于有两个或两个以上的解决方案，需要给出权衡各解决方案的考虑因素，并最终决定采用哪个解决方案。

实施策略

描述该问题解决方案的具体实施步骤，需要协调的资源，以及需要花费的时间等各个因素。
